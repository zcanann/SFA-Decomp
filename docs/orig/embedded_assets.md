# Embedded asset regions

Notes on `.data` regions of the retail `main.dol` that are **embedded binary assets**
rather than compiled C. dtk reconstructs their bytes byte-for-byte at build time from the
user-supplied disc under `orig/` (output lands in the gitignored `build/`); their bytes are
**never committed to this repo** (see *Why it isn't in source* below). They are catalogued
here so each reads as *identified and out-of-scope*, not as an unattributed gap in the
match. This mirrors `developer_artifacts.md`'s aim of turning anonymous blobs into named,
understood anchors.

## `0x802CC6A0`–`0x8030C6A0` — boot / loading-screen texture set (256 KB)

- **Identity (confirmed from code):** the game's **boot logo texture set** — three GameCube
  logo textures laid out back-to-back: the **Nintendo**, **Rareware**, and **Dolby Surround
  Pro Logic II** logos. Read at runtime by `initLoadingScreenTextures()` in
  `dll_0032_titlescreeninit.c`, which walks three `LoadingScreenTexture` records at
  `OSGetArenaHi() - 0x40000` (the arena copy of this region). Each record is a **0x60-byte
  header** (`width`@0x0A, `height`@0x0C `u16`; `format`@0x16 `u8`; pixels at `+0x60`) followed by
  the tile-swizzled image; `GXInitTexObj` is called with the header's own width/height/format, so
  the dimensions are **data-driven, not hardcoded** — which is how they were recovered exactly.
- **The three textures** (address = `0x802CC6A0` + offset; each = 0x60 header + image):

  | # | logo | width×height | format | header addr | record size | proposed symbol |
  |---|---|---|---|---|---|---|
  | 0 | Nintendo | 376×104 | IA8 | `0x802CC6A0` | `0x131E0` | `gLoadingScreenTextures` (base) |
  | 1 | Rareware | 128×160 | RGB5A3 | `0x802DF880` | `0xA060` | `gRarewareLogoTexture` |
  | 2 | Dolby Pro Logic II | 180×92 | CMPR (DXT1) | `0x802E98E0` | `0x22E0` | `gDolbyProLogic2Texture` |

  IA8/CMPR carry only intensity or a limited palette; the in-game yellow/gold is a render-time
  **TEV tint**, not stored in the texture. The three records fill `0x802CC6A0`–`0x802EBBC0`
  (`0x1F520`); everything after is `0xFF` padding.
- **Current placeholder:** the auto object `auto_07_802CBE94_data`
  (dtk labels `lbl_802CC6A0`, `lbl_802D808E`, `lbl_802E802E`, `lbl_802F808D` — the last three are
  mid-texture reloc-target artifacts, not real boundaries; superseded by the symbols above).
- **Proposed name:** `gLoadingScreenTextures` for the base (matching the code's own
  `LoadingScreenTexture` type / `initLoadingScreenTextures()`), plus the per-logo symbols in the
  table. `pi_dolphin.c:videoInit` hands this base to `GXInit` as the GPU FIFO — the region is
  reused as the write-only FIFO once the boot logos have been displayed.
- **Size:** `0x40000` (262,144 bytes) — three texture records (`0x1F520`) padded with `0xFF` to
  the `0x40000` GX-FIFO size.

### Runtime role
`videoInit()` copies the 256 KB to arena-top, then passes the same region to
`GXInit(base, 0x40000)` as the GPU command FIFO — the standard GameCube RAM-reuse pattern:
the initialized `.data` (the boot logos) is consumed at boot, and the RAM is then repurposed
as the write-only FIFO. `logGpuHang()` / `gpuErrorHandler()` also index diagnostic format
strings via `base + 0x4002c`.

### Note on the apparent "relocations"
dtk emits ~348 spurious `.4byte fn+offset` "relocations" inside the texture data. These are
false positives — mid-function targets at irregular stride, i.e. the analyzer matching
tiled/compressed image bytes against the `0x80xxxxxx` code-address range — not a real pointer
table.

### Why it isn't in source
It is an embedded copyrighted asset. Reproducing its 262,144 bytes in any form — a C array,
an assembler `.incbin`, or an extracted `.bin` — would redistribute Nintendo/Rare content in
a public repository, the one line a decompilation must not cross. Instead dtk extracts it
from the user's own disc at build time, reconstructing the DOL byte-perfectly with nothing
copyrighted committed.

**Status: identified, extracted, out-of-scope for source decompilation — complete.**
It contributes 0 to `matched_data` by design (the metric excludes `autogenerated` objects);
this is the correct outcome for an asset region, not outstanding work.

### Code-like edges (in scope — recovered as real C)
The compiler switch-jumptables and string literals bracketing the buffer are genuine source
and are attributed to their owning units, not part of the asset:
- `0x802CBE94`–`0x802CC6A0`: switch jumptables + asset-path/format strings
  (owners: `objprint_dolphin.c`, `pi_dolphin.c`).
- `0x8030C858`–`0x8030CF04`: inflate/deflate tables, a GX `IndTexCoordScale` table, and an
  OS debug format string (owner: `pi_dolphin.c`).
