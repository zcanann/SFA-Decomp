# Investigation: shared vs local int→float conversion magic (`lbl_803E3070`/`3068`)

**Question (from the kaldachom confound):** why do a few objects reference a
*shared external* int→float magic constant while every other build pools a
*local* copy — and is it fixable?

## What the constants are
- `lbl_803E3070` = `0x4330000080000000` (2^52 + 2^31) = **signed** int→double magic,
  emitted for `(f32)(int)x` / `(f64)(int)x`.
- `lbl_803E3068` = `0x4330000000000000` (2^52) = **unsigned** int→double magic,
  emitted for `(f32)(u32)x`.
These are compiler-generated by the MWCC backend for the standard PPC int→float
idiom (`xoris; stw; lis 0x4330; stw; lfd; fsubs`). They are NOT user literals.

## The facts gathered
1. **Only 4 objects reference the shared external symbol**, all in the **same
   (campfire) DLL**: `dll_00D5_kaldachom`, `dll_0129_campfire`, `cf_doorlight`,
   `dll_131`. Every other object (and every current decomp build, including these
   4) pools the magic **locally** (anonymous `.sdata2` `@N` entries).
2. **The conversion is byte-identical except the magic's reloc symbol.** Target:
   `lfd f1, lbl_803E3070@sda21(r2)`; ours: `lfd f1, @87@sda21(r2)`. Same 8-instr
   sequence otherwise. So it is purely *which symbol the load relocates against*.
3. **No compile-flag difference** — kaldachom's flags == a normal local-pooling
   DLL's flags, exactly.
4. **No source declaration** of the magic in any sibling `.c`. MWCC's backend
   generates it; it does not consult source-level `extern` declarations, so
   declaring/using `extern f64 lbl_803E3070;` cannot redirect the auto-conversion.
5. **`symbols.txt` mistypes both as `data:string size:0x3`** — they are 8-byte
   doubles. A separate data-side bug (the split would emit wrong-size data).

## Conclusion: NOT a per-function source fix
MWCC's automatic int→float conversion **always materializes a local `.sdata2`
magic**. There is no clean-C construct that makes the auto-conversion reference an
external shared symbol (a manual union-trick conversion referencing `extern
lbl_803E3070` would not reproduce the auto-sequence exactly). The 4 campfire
objects got external references because the **original campfire DLL externalized/
shared these constants at build time** — i.e. the magic lived once in the DLL's
shared `.sdata2` and the code objects imported it. That is a property of the
original DLL's compile/link structure, not reproducible by compiling one `.c`.

## Why this is (almost certainly) benign, and the real fix
- It is an **object-level** divergence in the relocation *target symbol*, with
  identical code bytes. In the final **linked** binary the load resolves to the
  same `.sdata2` address regardless of whether the symbol was local or shared, so
  it very likely does **not** affect the linked match — only objdiff's per-object
  symbol comparison flags it.
- **Recommended fix is on the tooling side, not the source:** teach the diff/
  matching to treat a local int→float magic pool entry (`@N` whose `.sdata2`
  value is `0x43300000_80000000` / `0x43300000_00000000`) as equivalent to the
  shared `lbl_803E30{70,68}` reference. That removes a whole class of false
  object-level mismatches (every `(f32)(int)` in these DLLs) without touching code.
- **Also worth doing (orthogonal):** fix the `symbols.txt` mistype
  `lbl_803E3070`/`3068` from `data:string size:0x3` → `data:double size:0x8`, so
  the data split emits the correct 8-byte constants.

## Bottom line
The campfire-DLL functions are blocked from a clean object-level 0 **only** by this
shared-vs-local magic-constant artifact — a build/link-structure property plus a
symbols.txt mistype, not a codegen or register issue. The compiler-pass model
(Coloring/Scheduler/CSE) still fully explains their *code*; this is a data/tooling
layer. Don't chase it with source edits.
