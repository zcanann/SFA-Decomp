# mwcc_re — static decompilation of MWCC GC/2.0 backend passes

**Goal.** Stop black-box guessing at what C produces a given asm shape. Instead,
*read the compiler's actual decision logic* by statically decompiling the
register allocator + scheduler from `mwcceppc.exe`, so register allocation and
instruction scheduling become **deterministically predictable** for any input.

This replaces the earlier *dynamic* (gdb-under-wibo) probing effort, which could
only ever produce observations, never the decision rules — every coalescer/priority
claim from that route ended up retracted. The blog post (`cracking-fn-801b3de4`)
is the cautionary tale: months on one function because nobody could state the rule.

## Why this is tractable (and was not, before)
1. `mwcceppc.exe` is a **2 MB i386 PE that `objdump` reads natively** (`pei-i386`).
   No Ghidra/IDA needed: `objdump -d --start-address=… --stop-address=…` gives
   flat-VA disassembly directly (image base `0x400000`, no ASLR).
2. The binary **kept its original source filenames** in `assert(file,line)` call
   sites, so every backend pass is a named, bounded TU. The map is already built:
   see `assert_map_GC2.0.txt` (generated by `tools/mwcc_re/mwcc_assert_map.py`).
3. The matching-relevant passes are **tiny**: the whole register-allocation +
   scheduling core is ~70 functions across 6 small TUs (sizes below).

## The passes that matter (and their size)
| TU | band | funcs | role |
|----|------|------|------|
| `Scheduler.c` | `0x508100-0x508680` | 5 | list-scheduler dependency DAG (`#pragma scheduling`) |
| `Coloring.c` | `0x508680-0x509010` | 8 | priority graph-coloring register allocator |
| `ValueNumbering.c` | `0x509010-0x50a6e0` | 15 | web numbering / copy folding (`opt_common_subs`) |
| `InterferenceGraph.c` | `0x57b680-0x57c290` | 3 | builds the interference graph |
| `SpillCode.c` | `0x57c290-0x57d2a0` | 15 | spill insertion |
| `RegisterInfo.c` | `0x4d0150-0x4d11b0` | 18 | per-class register tables (the `0x5e9b04` etc. globals) |
| `IroCSE.c` | `0x46a360-0x46bef0` | 17 | earlier CSE (the *other* place a copy dies) |

Peephole (`#pragma peephole`, the single most-used pragma, 358 sites) has **no
`Peephole.c` assert TU** — it is inlined into the PCode emit TUs
(`PCodeListing.c`/`PCodeUtilities.c`/`PCodeAssembly.c`); locating it is on the worklist.

## Method (per pass)
1. `objdump -d` the band → `disasm/<TU>.objdump.txt` (checked in as evidence).
2. Recover struct offsets from the access patterns; cross-check globals against
   the (validated) dynamic-probe offsets where they exist (web `+0x14`=reg,
   `+0x16`=flags — kept in `dynamic_findings.md`).
3. Hand-decompile to annotated C in `recovered/<TU>.c`. **Honesty rule (carried
   over from the prior effort): mark every inferred name `[inf]`; never state a
   rule the disassembly doesn't show. A retraction beats an over-claim.**
4. Once a pass reads cleanly, derive the **plausible-C levers** it implies and
   fold them back into `CLAUDE.md`'s "few MWCC facts" list — each one now backed
   by the actual code path, not a black-box correlation.

## Tools
- `tools/mwcc_re/mwcc_assert_map.py <mwcceppc.exe>` → regenerates `assert_map_GC2.0.txt`
  (needs `pefile` + `capstone`, both already installed).
- `objdump -d --start-address=0xADDR --stop-address=0xADDR build/compilers/GC/2.0/mwcceppc.exe`
  is the workhorse; data/strings via `objdump -s --start-address=… --stop-address=…`.
