# Content-homing work queue

Authoritative per-file content-coherence queue for every dll-hosting
unit. Generated by `python3 tools/content_homing_audit.py` (READ-ONLY —
the tool edits nothing but this doc). Re-run after every content-homing
wave. Built on the retail TU model (`dll_boundary_audit.load()`), the
canonical-name manifest (`dll_naming_manifest.md`), and the lane-homing
census (`cf_lane_homing.py`).

Each unit gets: resident descriptors (dll id + retail name), resident-fn
count, an fn-PREFIX census (the families living in the file), a VERDICT,
an EFFORT class, and — for non-OK rows — the proposed name / carve plan.
An **anomaly** is an fn-family prefix matching NEITHER the filename stem
NOR any resident descriptor's retail/canonical name — the mechanical
catch for the spellstone-in-prisonuncle content-drift class.

## Partition plan (concurrent execution batches)

Actionable items packed into address-ordered batches over DISJOINT
`splits.txt` / `configure.py` regions. Each batch is a CONTIGUOUS address
run, so multiple agents can run in separate worktrees and edit the global
files without colliding — assign one batch per worktree, land
address-edge-first within a batch (the resplit tool's linear-link-order
rule). Batches are independent; run them in any order or all at once.

Batches: 12. Region-disjoint: NO OVERLAP — verified (each batch owns a contiguous address interval; no item's footprint straddles a batch boundary).
