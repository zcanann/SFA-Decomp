# DLL canonical-naming manifest

Authoritative dll-id -> canonical-filename mapping for the rename
campaign. Generated by `python3 tools/dll_canonical_names.py --regen`
(re-run after each boundary-surgery wave; READ-ONLY analysis — no
rename is executed by the tool).

Retail names come from OBJECTS.bin defs (name at def+0x91, 11-char
fixed field; handling dll id at def+0x50). Expansion status:
**COMPLETE** = name shorter than the 11-char field (cannot be
truncated); **CONFIRMED** = expansion/stem backed by symbols.txt
fn-name prefixes or descriptor symbol names; **GUESSED** = backed only
by an existing filename or a multi-def common stem; **RAW** = 11-char
name kept truncated, no evidence found.

Lane-dir rule: a unit already inside a `main/dll/<LANE>/` dir keeps
that dir (address-lane ownership beats name prefix — precedent:
`DR/dll_0149_cfwindlift.c`); otherwise the dir derives from the name
prefix (`WM_`->WM/ etc; `CC`/`SB` dirs do not exist yet —
create-listed, do not create until the first rename lands); names
with no lane prefix stay at `main/dll/` root.
